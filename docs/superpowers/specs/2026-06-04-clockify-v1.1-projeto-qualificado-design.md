# Clockify v1.1 — Resolução por (Projeto, Tarefa) + Histórico — Design

**Data:** 2026-06-04
**Autor:** vbjuliani (PG Consulting)
**Status:** Aprovado (design) — pendente plano de implementação
**Relacionado:** estende o plugin [[2026-06-03-clockify-plugin-compartilhavel-design]] (v1.0). Versão alvo: **1.1.0**.

## Problema

O plugin resolve tarefa **por nome global** (`build_payload._resolve_task`). Num workspace de
consultoria isso quebra: medido no workspace real (`PG Tech`), há **1055 tarefas mas só 132
nomes distintos** — 92% das tarefas têm nome repetido entre projetos (ex.: `01. Gestão do
Projeto` existe em 42 projetos). O guard de ambiguidade (v1.0) faz o `add` **bloquear** nesses
casos — correto para integridade, mas inviabiliza colegas multi-cliente: para eles, quase toda
tarefa é ambígua por nome.

Objetivo: permitir resolver tarefa por **`(projeto, tarefa)`**, com a atribuição do projeto
acontecendo **na conversa** do `/horas`–`/lancar` (a pessoa sabe o projeto do dia), e um
**histórico local** que aprende mapeamentos recorrentes e os sugere — reduzindo a fricção ao
longo do tempo.

## Decisões tomadas (brainstorming)

| Tema | Decisão |
|---|---|
| Mecanismo de atribuir projeto | **Seleção interativa por item/lote** no `/horas`–`/lancar`. A pessoa só lida com os poucos itens do dia, nunca com as 1055 tarefas. |
| Resolução | Por `(projeto, tarefa)` quando o projeto é informado; **fallback por nome único** quando não (retrocompatível — `Time IA` único continua funcionando). |
| Histórico | Arquivo local `~/.config/clockify-horas/history.json`. Match **exato** por descrição normalizada. Sem fuzzy. |
| Gravação do histórico | Automática no `add`, após cada lançamento criado com sucesso (determinístico, na CLI). |
| Sugestão | `clockify-horas suggest --description "..."` devolve o mapeamento histórico exato. O `/horas`–`/lancar` pré-preenche e a pessoa confirma. |
| Keyword | Via **override manual** (já existe). Promoção automática de padrão→override fica desenhada como **v1.2** (fora de escopo agora). |
| Navegação de tarefas | **Fora de escopo** (YAGNI). A pessoa nomeia o projeto do dia; erro claro se errar o nome. |

## Princípio guia

Manter simples. A resolução continua "nomes → IDs" (separação cérebro/IO intacta): o cérebro
(conversa) decide projeto/tarefa por item, a CLI resolve e grava de forma determinística e
testável. Nada auto-aplica sem o dry-run + confirmação que já existem.

## Componentes

### 1. Resolução `(projeto, tarefa)` — `entries.py`

`_resolve_task(task_name, metadata, project_name=None)`:
- **Com `project_name`:** resolve `project_name` → `project_id` (via `metadata.projects`; erro
  "projeto não encontrado" se ausente); acha `task_id` onde `(project_id, task_name)` está em
  `metadata.tasks` (erro "tarefa T não existe no projeto P" se ausente). Retorna `(project_id, task_id)`.
- **Sem `project_name`:** comportamento atual — match por nome; único → retorna; ambíguo → erro
  listando os projetos **e sugerindo qualificar com o projeto**; nenhum → erro.

`build_payload(entry, metadata)` passa `entry.project_name` para `_resolve_task`.

### 2. Modelos — `models.py`

- `TimeEntry`: novo campo `project_name: str | None = None`.
- (Resto inalterado — `Metadata` já tem `projects` (nome→id) e `tasks` ((projId, nome)→id).)

### 3. Config — `config.py`

- `Defaults`: novo campo `project: str | None = None` (opcional).
- `Override`: novo campo `project: str | None = None` (opcional).
- `load_defaults`/`load_overrides`: leem `project` quando presente; ausente → `None`
  (retrocompatível com configs v1.0).
- `from_event(event, defaults)` propaga `project_name=defaults.project`.

### 4. Histórico — novo módulo `history.py`

- `history_path()`: `~/.config/clockify-horas/history.json` (mesma raiz XDG/`%APPDATA%` do
  `config_path`, arquivo separado para não poluir o `config.json` editável à mão). Só local,
  nunca no repo. Escrita com `chmod 600` em POSIX.
- Schema: `{ "<descrição normalizada>": {"project_name": str|null, "task_name": str,
  "tag_names": [str], "billable": bool} }`. **As chaves espelham o item do JSON do `add`**
  (`project_name`/`task_name`/`tag_names`/`billable`), para a sugestão mapear direto sem
  renomear campo (evita `KeyError` no `add`). Normalização: `description.strip().lower()`.
  Upsert sobrescreve (última gravação vence — sem timestamp; YAGNI).
- `read_history()` / `write_history()` / `record_entry(description, project, task, tags, billable)`
  / `suggest_for(description) -> dict | None`.

### 5. CLI — `cli.py`

- `config set --project "..."` → grava `defaults.project`.
- `config add-override --project "..."` → grava `override.project` (opcional).
- `add`: cada item do JSON pode trazer `project_name` (opcional) → `TimeEntry.project_name`.
  **Após criar cada lançamento com sucesso**, chama `history.record_entry(...)`. Em
  `--dry-run` **não** grava histórico. Em falha parcial, só os itens já criados entram no histórico.
- `suggest --description "<texto>"` (novo subcomando): imprime JSON `{project, task, tags, billable}`
  do match exato, ou `{}` (exit 0) se não houver.
- `config show`: exibe os campos `project` (sem redação; não é segredo).

### 6. Slash commands — `commands/horas.md`, `commands/lancar.md`

- Ao montar candidatos (eventos da agenda / atividades ditadas), para cada item rodar
  `clockify-horas suggest --description "<descrição>"`. Se houver sugestão, **pré-preencher**
  projeto/tarefa/tag/faturável e dizer "da última vez foi projeto X / tarefa Y — mantenho?".
- **Precedência ao montar o candidato:** override declarado (keyword `match`) **>** sugestão do
  histórico **>** default. A pessoa pode trocar qualquer item ("esse é do projeto Z").
- O JSON do `add` ganha `project_name` (opcional) por item — atualizar o exemplo de schema nos
  dois prompts. Itens sem `project_name` resolvem por nome único (retrocompat).
- Dry-run + confirmação continuam obrigatórios.

### 7. Skill `/clockify-setup` — `skills/clockify-setup/SKILL.md`

- No passo de Defaults, permitir que o default seja **opcionalmente** qualificado por projeto
  (`config set --project "..."`) — pular se a tarefa default for de nome único (caso do `Time IA`).

## Error handling

- **Projeto informado não existe** → erro claro "projeto P não encontrado" no `add` (return 1, sem POST).
- **Tarefa não existe no projeto** → "tarefa T não existe no projeto P" (return 1).
- **Nome ambíguo sem projeto** → erro atual, agora **sugerindo** qualificar com o projeto.
- **`suggest` sem match** → `{}` + exit 0 (não é erro).
- **`history.json` ausente/corrompido** → tratado como vazio (nunca quebra o `add`/`suggest`).

## Testes

- `_resolve_task`: com projeto (resolve dentro do projeto; projeto inexistente; tarefa fora do
  projeto); sem projeto (nome único resolve; nome ambíguo erra; inexistente erra).
- `config set --project` e `add-override --project` persistem; `load_defaults`/`load_overrides`
  retrocompat (sem `project` → None).
- `add`: grava histórico no sucesso; **não** grava em `--dry-run`; falha parcial só grava os criados.
- `suggest`: match exato retorna mapeamento; sem match retorna `{}`.
- `history`: normalização (trim/lowercase); upsert sobrescreve; `chmod 600` POSIX.
- Suíte v1.0 segue verde (campos `project`/`project_name` são opcionais).

## Plano de fases (visão)

1. **Núcleo de resolução** — `_resolve_task(project_name)`, `TimeEntry.project_name`, `add` lê
   `project_name`, erros. (TDD)
2. **Config qualificado** — `project` em Defaults/Override, `config set --project`,
   `add-override --project`. (TDD)
3. **Histórico** — `history.py`, `add` grava, `suggest` subcomando. (TDD)
4. **Conversa** — generalizar `/horas`, `/lancar` (suggest + precedência + schema com
   `project_name`); `/clockify-setup` projeto opcional.
5. **Release** — bump 1.1.0 em lockstep (plugin.json, marketplace, pyproject, __init__, test_smoke);
   `claude plugin validate`; atualizar README/MAINTAINER.

## Fora de escopo (YAGNI v1.1)

- Match fuzzy/keyword na sugestão (override manual cobre keyword).
- Promoção automática de padrão→override (desenhada para **v1.2**).
- Comandos de navegação (`projects`, `tasks --project`, `tasks --search`).
- Auto-aplicar sugestão sem confirmação.
- UI de gestão do histórico (é um JSON; limpar = apagar o arquivo).
- Timestamp/contagem no histórico.
